# Crea UX Restructure — Design Spec

**Date:** 2026-03-12
**Status:** Draft
**Context:** Emerged from a full UX audit of the Crea prototype using a 9-lens design framework (Context of Use, Progressive Disclosure, Visual Hierarchy, Emotional Design, Fitts's Law, Feedback & State, Information Density, Motivation Architecture, System Connection).

---

## Problem

The original 3-tab layout (Workouts, Nutrition, Check-In) had structural issues:

- **Workouts tab** was cluttered with non-workout content (metrics dashboard, mode toggle, phase banner, weekly targets) that buried the primary action: today's workout.
- **Nutrition tab** was entirely static — no AI integration, no personalization.
- **Check-In tab** tried to serve two purposes: a weekly data-collection event AND an always-available progress view. These are fundamentally different interactions.

## Solution: 3 Tabs + 1 Modal

### Tab Structure

| Element | Purpose | When it gets attention |
|---------|---------|----------------------|
| **Workouts** | Today's workout. What to do at the gym. | Daily (gym days) |
| **Lifestyle** | Nutrition targets, sleep, meal planning | Daily + Saturday (grocery prep) |
| **Progress** | AI-narrated progress story + evidence | Weekly (after check-in updates) |
| **Sunday Check-In** (modal) | 60-second weekly data collection | Sunday only |

---

## 1. Workouts Tab

### What stays
- **Header** with program name, week number, mode badge (Push/Maintain) — scrolls away with content, not fixed.
- **Day cards** (Mon–Sat) showing exercises, sets, reps, AI-suggested weights.
- **Today auto-expanded**, other days collapsed.
- **Done/not-done visual state** per day.

### What moves or is removed
- **Metrics dashboard** → removed from Workouts (lives in Progress tab as "Under the Surface" evidence card)
- **Mode toggle** → removed entirely (AI decides Push/Maintain, user doesn't override)
- **Phase banner** ("This is a Push Week...") → removed from tab. Becomes a push notification (6am + 3pm if app not opened) and lives in the check-in modal result. Separate spec needed.
- **Weekly targets** ("2/3 vigorous days done") → moved to Progress tab

### Post-Workout Logging Flow

**Design principle:** Zero cognitive load during the workout. Logging happens AFTER, not during.

1. User taps "I'm Done" on today's card.
2. A **post-workout modal** appears showing each exercise from today.
3. For each exercise, user rates the AI-suggested weight: **Too Light / Good / Too Heavy / Incomplete**.
4. Optional: free-text note field (e.g., "shoulder felt tight").
5. Submit → backend stores ratings → AI adjusts future weight suggestions.

**Key constraint:** User explicitly does NOT want to edit weights or reps line-by-line during the workout. The AI suggests, she executes, then she gives quick feedback afterward. Minimal thinking.

**API contract change:** The existing `POST /api/workout/complete` accepts `actual_weight_used`, `sets_completed`, and `reps_completed` per exercise. This changes:
- `actual_weight_used` → defaults to the AI-suggested weight (user doesn't enter it)
- `sets_completed` and `reps_completed` → auto-filled from the plan (assumed completed unless rating is "Incomplete")
- `weight_rating` remains the primary user input
- `notes` remains optional
- The request body simplifies to: `{ exercises: [{ exercise_name, weight_rating, notes? }] }` — the backend looks up the plan to fill in the rest.

## 2. Lifestyle Tab

### v1 Scope

#### Nutrition Targets Card
- Displays the `nutritionJson` from the current `weeklyPlan` (already generated by the plan generator during check-in).
- Shows: daily calorie target, protein (g), and any AI-generated focus notes (e.g., "carb timing around workouts").
- The plan generator's prompt currently hardcodes "1750 cal, 135g protein" as a baseline — this should be updated to make targets dynamic based on mode, body weight, and activity level. Push weeks get higher protein/calories; Maintain weeks are moderate.
- Served via existing `GET /api/plan/current` (the `nutritionJson` field).

#### Sleep Card
- Single card showing: target sleep hours and suggested bedtime.
- Data source: Garmin sleep averages from `garminMetrics` table. The AI uses the user's recent sleep average to suggest a realistic +15-30 min improvement target.
- Generated as part of the weekly plan's nutrition/lifestyle output (add a `sleepTarget` field to the plan generator's output).

#### Grocery/Meal Planning Placeholder
- A card with "Coming Soon" text and a brief description: "Grocery lists and recipes based on your nutrition plan."
- No functionality in v1 — purely a placeholder to establish the tab's future direction.

### What's removed
- **"5 Non-Negotiable Rules"** — removed. Static rules don't belong in an AI-driven app.

### Future (separate spec)
- PCOS-aware meal planning and recipes
- Grocery list generation linked to nutrition targets
- Instacart/DoorDash/Uber Eats integration
- Saturday notification for grocery prep

## 3. Progress Tab

### Design Philosophy
**Story first, data second.** The AI writes a narrative about the user's progress. The data is evidence that supports the story — not the other way around.

This tab answers four questions the user cares about:
1. "Is this working?" — proof that effort produces results
2. "What's happening in my body?" — invisible health trends (HRV, sleep, stress, resting HR)
3. "Look what I can do now." — strength gains, capability milestones
4. "Tell me my story." — AI-narrated, not dashboards

### Layout (top to bottom)

#### AI Progress Narrative (hero section)
- Generated during the Sunday check-in flow (Step 3, alongside plan generation).
- Stored in a new `progressNarrative` text column on `weeklyPlans` (generated alongside the plan, so it has access to all the same data).
- Served via `GET /api/progress/narrative` (returns the most recent week's narrative).
- 3 paragraphs written by the AI in second person ("Your resting heart rate has dropped...").
- Reads like a trainer talking to her — warm, specific, evidence-based.
- References specific numbers but weaves them into a story.
- Ends with a forward-looking insight ("Keep protecting your sleep. It's driving everything else.").
- The AI prompt receives: all Garmin metric trends, exercise log history with weight progressions, check-in history (energy/motivation), photo analysis results, and mode history.

#### "What You Can Do Now" (evidence card)
- Shows the user's **top compound lifts** with current best weight and delta from first logged session.
- Exercises shown: the exercises that appear most frequently in her plans (top 4-6 by frequency).
- "Current weight" = the most recent weight logged with a "Good" or "Too Light" rating (excludes "Too Heavy" and "Incomplete").
- "Change from start" = current weight minus the weight from her first logged session for that exercise.
- Served via `GET /api/progress/strength-gains` (computes from `exerciseLogs` table).
- Pink/coral accent color (`#f5576c`).

#### "Under the Surface" (evidence card)
- Health metrics trends: Resting HR, HRV, Sleep avg, Stress avg.
- Each shows current value (most recent week's average) and direction of change (vs. first week's average).
- Data source: `garminMetrics` table, aggregated by week.
- Served via existing `GET /api/metrics/latest` (may need a companion `GET /api/metrics/trends` endpoint for deltas).
- Blue accent color (`#5B7FBF`).

#### "Your 12 Weeks" (journey visualization)
- Minimal bar chart showing all 12 weeks of the current program cycle.
- Color = mode (gold = Ramp-Up, pink gradient = Push, green = Maintain, gray = upcoming).
- Height = intensity level.
- Current week highlighted with border.
- Legend at bottom.
- **12-week boundary:** The program runs in 12-week cycles. At week 13, a new cycle begins (week numbers reset, new goals can be set). The journey visualization always shows the current cycle. Historical cycles are preserved in the database but not displayed in v1.

## 4. Sunday Check-In Modal

### Trigger
- Appears Sunday evening (time TBD).
- Cannot be dismissed until completed.
- Blocks app usage until done.
- **Missed Sunday:** If the user opens the app Monday or later without completing the check-in, the modal still appears and blocks until done. The check-in is required before any new week's data is generated.
- **Partial state:** If the app is force-quit mid-check-in, the modal restarts from the beginning on next open. No partial state is saved (it's only 2-3 taps, not worth the complexity).
- **Deadline:** None. The check-in waits until completed. The previous week's plan remains active until the new one is generated.

### Flow (60 seconds total)

**Step 1: How are you feeling? (10 seconds)**
- Energy this week: 1–5 tap scale
- Motivation: 1–5 tap scale
- Optional free-text note ("shoulder felt tight Thursday")

**Step 2: Progress photo (10 seconds, optional)**
- Snap a photo OR upload from saved photos (camera roll)
- "Same lighting, same pose" guidance text
- "Skip for now" option

**Step 3: AI works (5 seconds — loading state)**
- Diamond logo pulses
- "Building your week..." with status lines:
  - Pulling your Garmin data
  - Analyzing recovery signals
  - Generating your plan

**Step 4: Result (the payoff)**
- Mode badge: "PUSH WEEK" or "MAINTAIN WEEK"
- One-line subtitle: "You're ready to go hard." or "Recovery is the priority this week."
- Two CTAs:
  - **See My Workouts →** (navigates to Workouts tab with new week loaded)
  - **See My Lifestyle Plan →** (navigates to Lifestyle tab with updated nutrition)
- No trainer summary paragraph — the mode + subtitle is enough.

### What happens invisibly (backend)
- Auto-pull Garmin metrics from Gmail (HRV, sleep, body battery, stress, steps, resting HR)
- Auto-analyze progress photo vs. goal references + last check-in
- Auto-decide Push/Maintain/Rampup mode via recovery assessment
- Auto-generate 7-day workout plan with personalized weights
- Auto-generate nutrition targets
- Store everything for trend tracking
- Update Progress tab narrative

## 5. Push Notifications (separate spec needed)

- **Phase notification**: Sent at 6am and again at 3pm IF the user hasn't opened the app yet that day.
- Content: mode badge + focus for the week (same content all week).
- Example: "PUSH WEEK — You're lifting heavier this week. 3 strength sessions, 2 cardio."
- Replaces the old phase banner that was on the Workouts tab.

## 6. API Changes Summary

### Existing endpoints (no change)
- `GET /api/plan/current` — serves Workouts tab day cards + Lifestyle tab nutrition targets
- `GET /api/metrics/latest` — serves "Under the Surface" card (current values)
- `POST /api/checkin` — accepts energy, motivation, notes, photo

### Modified endpoints
- `POST /api/workout/complete` — simplified request body: `{ exercises: [{ exercise_name, weight_rating, notes? }] }`. Backend auto-fills weight/sets/reps from plan.

### New endpoints
- `GET /api/progress/narrative` — returns the AI-generated progress story (stored on `weeklyPlans`)
- `GET /api/progress/strength-gains` — computes per-exercise current weight + delta from start
- `GET /api/metrics/trends` — returns metric deltas (current week avg vs. first week avg)

### Schema changes
- Add `progressNarrative` text column to `weeklyPlans` table
- Add `sleepTargetHours` and `sleepTargetBedtime` fields to plan generator output (stored in existing `nutritionJson` or a new `lifestyleJson` column)

### Deprecated
- `trainerMessage` field on `checkIns` — still stored by the backend but no longer displayed in the UI. The check-in result screen shows only mode + subtitle. The narrative moves to the Progress tab. Field can be removed in a future cleanup.
